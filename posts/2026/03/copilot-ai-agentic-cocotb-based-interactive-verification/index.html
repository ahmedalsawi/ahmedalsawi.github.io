Copilot AI Agentic Cocotb-based Interactive Verification

March 10, 2026 5-minute read
Llm • Cocotb

Here is a crazy idea! We all used Copilot to generate verilog RTL or systemverilog tests. Now, what if we give an agent direct access to read signals, drive ports, wait for delays, run a few clock cycles. Essentially, giving full access to an agent and asking it to go to town on the RTL. I hear you saying “why?”. My usual reply is what else is there to do on a such lovely night!

But really, the advantage could be that we don’t need to generate code (that might have syntax errors or much worse semantic errors). The idea is creating an MCP server that starts cocotb in the background and uses ZeroMQ to send commands from yet another MCP tool to this background cocotb simulation. Again, why? Because cocotb doesn’t work with asyncio. Technically I could use HTTP but why would I do that when we have ZeroMQ.

Full disclosure I wrote this from scratch with a Langchain agent but I am too lazy to rewrite it with MCP for copilot so I asked copilot to do it for me.

The Copilot agent communicates with an MCP server that manages cocotb simulations, which controls Verilog RTL.

Installing the MCP server for copilot is easy enough. Just add a configuration file at .vscode/mcp.json

{
	"servers": {
		"cocotb-simulator": {
			"type": "stdio",
			"command": "${workspaceFolder}/.venv/bin/python3",
			"args": ["${workspaceFolder}/src/cocotb_mcp.py"]
		}
	},
	"inputs": []
}

The cocotb test file interactive_test.py contains a simple test that creates a ZeroMQ server to receive and execute simulation commands:

async def interactive_test(dut):
    """
    Interactive test that waits for commands via ZeroMQ
    
    This test creates a ZeroMQ server that listens for commands to control
    the simulation. The simulation will run indefinitely until a 'stop' command
    is received.
    """
    
    dut._log.info("=== Starting Interactive Simulation ===")
    dut._log.info("Waiting for commands on ZeroMQ port 5555...")
    dut._log.info("Commands: set_signal, get_signal, wait_ns, wait_clocks, start_clock, stop")
    
    server = SimulationServer(dut, port=5555)
    
    try:
        await server.run()
    finally:
        server.close()
    
    dut._log.info("=== Interactive Simulation Ended ===")

Verify the RTL at examples/adder.v using cocotb interactive simulation from cocotb-simulator mcp.

Conclusion Link to heading

Is this better than generation tests (either generating cocotb tests or SystemVerilog UVM tests)? My answer is “who cares!”. To be honest, It’s probably a waste of tokens to let an agent do these repeatedly and it will be slower than generating tests and running them. But it’s surely fun to see :)
